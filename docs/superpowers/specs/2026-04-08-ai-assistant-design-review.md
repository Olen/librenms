## Critical Concerns

### 1. Proactive Monitoring in v1 is High Risk

> **Resolution: Design updated.** Proactive monitoring is now opt-in and disabled by default.

The monitoring engine will generate false positives from day one because the pattern store starts empty. Every recurring maintenance window, expected failover, and scheduled backup-induced I/O spike will be flagged as anomalous.

Additionally, cost explosion is a concern: every poller cycle (typically 5 minutes) with "notable changes" triggers a full LLM call with 30 minutes of event context. On busy networks this could be every single cycle. Hitting the daily budget by noon means interactive chat goes dark too.

**Recommendation:** Consider a "learning-only" mode where the system observes and builds patterns but doesn't inject alerts until an admin enables injection after reviewing initial patterns.

### 2. Full Access for Monitoring Engine Bypasses RBAC

Section 3 states that when called from the monitoring engine (no user context), full access is granted. The monitoring engine then injects alerts via existing transports. If those transports go to shared channels (Slack, IRC), the alert message text — generated by the LLM from full-access data — is visible to everyone, potentially leaking information about devices that specific users shouldn't know about.

The design's "defense in depth" RBAC only covers interactive chat. The monitoring → alert injection → transport pipeline has no filtering layer.

**Recommendation:** AI-injected alerts should go through the same transport filtering that normal alerts use, ensuring device-group-based visibility is respected.

### 3. LLM-Generated Alert Suppression is Dangerous

The pattern store allows `approved` + `suppression` patterns to prevent alerts. Even with human review:

- The reviewer sees a text description written by the LLM ("Backup job causes I/O spike daily 00:00-00:30"). They can't verify this is actually true without deep investigation.
- If the LLM mischaracterizes a correlation, the admin may approve a suppression rule that hides real incidents.
- The `schedule` field is a free-text string, not a structured cron expression. The LLM must interpret it each flush cycle, making the suppression boundary fuzzy and non-deterministic.

**Recommendation:** Suppression patterns should map to structured, verifiable conditions (device IDs + time windows as cron expressions + specific alert rule IDs), not free-text descriptions that the LLM interprets at runtime.

## Significant Concerns

### 4. EventListenerHook is Synchronous and Unbounded

The `handleEvent()` method is called inline at dispatch points (alert firing, polling, syslog ingestion). If the AI plugin does anything slow in `handleEvent` — even just a database write — it adds latency to the poller and alert pipeline.

The hook interface provides no timeout or async contract. A badly-behaved plugin implementing this hook could stall the entire poller.

**Recommendation:** Document that `handleEvent` must be non-blocking. Consider a queue-based dispatch model or at minimum a try/catch with timeout in PluginManager's dispatch loop.

### 5. Context Window Exhaustion

The design sends full conversation history + system prompt to the LLM on every message. With a system prompt that includes context snapshot, tool listings, learned patterns, and permission scopes, plus up to 50 messages of history (which can include large tool call results), smaller models (especially local Ollama with 4-8k context) will overflow.

The `getMaxContextTokens()` method exists on the provider but is never referenced in the conversation flow. History trimming is by message count, not token count.

**Recommendation:** Implement token-aware history trimming. Before sending, estimate total tokens and drop oldest messages (or summarize them) to fit within `getMaxContextTokens()` minus a reserve for the response.

### 6. No Streaming, But Long Latency

A chat query triggering multiple tool calls goes through an iterative loop (up to 10 rounds of LLM call → tool execution → LLM call). This could easily take 15-30 seconds with API latency.

Users staring at "Thinking..." for 30 seconds with no feedback will assume it's broken. On IRC there's no typing indicator at all.

**Recommendation:** At minimum, send intermediate status messages ("Checking device status...", "Looking up alert history...") as tool calls execute. Plan for streaming in v1.1.

### 7. The "Ring Buffer" Doesn't Exist

Section 6 and the architecture diagram reference a "ring buffer," but the implementation description queries events from the database each cycle. There is no in-memory ring buffer.

The naming is misleading and could confuse implementers. Performance is entirely dependent on database query speed over event/syslog tables, which on busy networks can be very large.

**Recommendation:** Rename to "sliding window query" in the spec to match the actual design, or implement an actual in-memory event buffer if performance requires it.

### 8. No Retry or Error Handling Strategy

The spec doesn't address:
- LLM API rate limiting (429 responses)
- LLM API downtime
- Malformed JSON in structured output
- Tool execution exceptions mid-loop

For monitoring (which runs automatically), a crash in the monitoring flush shouldn't prevent the next cycle, and repeated API failures should back off exponentially, not hammer the endpoint.

**Recommendation:** Define retry strategy with exponential backoff, circuit breaker for repeated failures, and graceful degradation (e.g., skip monitoring flush on API error, but keep chat available to respond with cached context).

## Minor Concerns

### 9. IRC Session Model is Fragile

Sessions are per user-channel combination. A user asking in `#network` who continues in a DM gets a fresh session with no context. Two users in the same channel have isolated sessions but interleaved bot responses, which could be confusing.

### 10. Pattern Confidence Score is Meaningless

LLMs are notoriously bad at self-calibrating confidence. A 0.9 confidence score from the LLM is not actually 90% likely to be correct. This number will mislead admins into trusting high-confidence patterns without scrutiny.

**Recommendation:** Replace or supplement with `occurrences` count as the primary trust signal. A pattern seen 15 times is more trustworthy than one the LLM rates at 0.95 confidence.

### 11. No Migration Path for Plugin Data

Six new database tables are created. The spec doesn't address:
- Plugin uninstall (tables dropped or orphaned?)
- Plugin upgrades (migration versioning within plugin context)
- LibreNMS core upgrades that change models the tools depend on

### 12. `ai.temperature` is Global

One temperature setting for all use cases. Chat benefits from slightly higher temperature (natural responses), while monitoring analysis needs low temperature (deterministic structured output). The spec acknowledges this but provides only one config key.

**Recommendation:** Use separate temperature settings per context: `ai.temperature_chat`, `ai.temperature_monitoring`, `ai.temperature_reports`.

---

## What's Good

- **Plugin architecture with generic hooks** — clean separation, good for upstream acceptance
- **Defense-in-depth RBAC for chat** — filtering data before it reaches the LLM rather than relying on prompt instructions
- **Cost tracking and budget caps** — essential for any LLM integration
- **OpenAI-compatible provider first** — pragmatic choice covering the most backends
- **Context snapshot injection** — smart optimization avoiding tool calls for common questions
- **Pattern store with human review lifecycle** — the right trust model for learned knowledge
